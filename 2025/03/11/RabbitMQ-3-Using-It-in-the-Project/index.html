為什麼要抽象 RabbitMQ Client

在最基本的 RabbitMQ 例子中，我們通常會直接使用 pika 來連接伺服器、宣告佇列並發送訊息。然而，這樣的程式碼往往忽略了許多關鍵細節，例如錯誤處理、連線管理、訊息持久化等。如果我們將這些細節補充完整，程式碼就會變得更加複雜。例如，下面的範例加入了錯誤處理：

雖然這樣的程式碼已經比原始版本更加穩定，但在實際開發中，RabbitMQ 會被廣泛應用於不同的模組，如 訂單處理（Order Service）、支付服務（Payment Service）、訊息通知（Notification Service） 等。這意味著：

  1. 每個模組都需要處理 RabbitMQ 連線 —— 但這些連線邏輯其實是相同的。
  2. 錯誤處理邏輯可能被重複實作 —— 每個地方都要加上相同的錯誤處理，增加了維護成本。
  3. 代碼難以擴展 —— 如果未來想要新增其他功能（例如支援不同的 RabbitMQ 交換機類型、訊息重試機制等），會導致每個使用 RabbitMQ 的模組都需要修改。

為了避免這些問題，我們可以將 RabbitMQ 抽象為一個單獨的 Client 類別，這樣的設計帶來以下好處：

  1. 統一管理 RabbitMQ 連線：所有需要 RabbitMQ 的地方都可以直接使用這個類別，不需要每次都寫重複的連線邏輯。
  2. 內建錯誤處理：錯誤處理邏輯集中管理，不需要每個地方都寫一遍。
  3. 提升可維護性：如果未來 RabbitMQ 的設定需要修改（例如更換主機、調整認證機制），只需要改動這個類別，而不會影響其他模組。
  4. 方便擴展：如果未來需要支援不同的佇列機制（例如 Fanout、Topic、Headers 交換機等），可以直接在這個類別內擴展，而不用修改現有的業務邏輯。

OrderEvent

在此專案中，我們不再像之前範例那樣將 producer 和 consumer 分成兩個獨立的 .py 檔案。相反地，我們將所有訂單服務所需的 producer 和 consumer 功能都封裝在類別中，主要包含以下功能：

  1. 建立 Exchange & Queue
  2. 發訊息到 Queue
  3. 關閉 Queue

📌 Producer（生產者）

1️⃣ OrderEventProducer

OrderEventProducer 是負責發送訂單相關事件的生產者，它會將訂單狀態的變更發送至 RabbitMQ Exchange（ORDER_EXCHANGE），包括：

  • Publish Order Created（訂單創建事件）
  • Publish Payment Processed（付款處理事件）
  • Publish Order Shipped（訂單發貨事件）

這些事件會根據不同的 Routing Key（路由鍵），被分發到不同的 消息佇列（Queue） 內，以便消費者（Consumer）來處理。

📌 RabbitMQ（交換機與佇列）

2️⃣ 交換機（Exchange）

在這個架構中，我們使用 ORDER_EXCHANGE 作為 RabbitMQ 交換機，它的主要作用是：

  1. 接收來自 Producer 的事件消息。
  2. 根據不同的 Routing Key（路由鍵），將消息分發到不同的佇列（Queue）。

交換機的類型

  • 這裡主要使用 Direct Exchange（直連交換機），即 Routing Key 必須 完全匹配，才能讓消息被路由到特定的佇列。
  • 另一種可能的方式是 Topic Exchange（主題交換機），允許透過 和 # 來進行 模式匹配，例如：
    • order.* 會匹配 order.created 和 order.shipped。

📌 Consumer（消費者）

OrderEventConsumer 是專門用來處理 RabbitMQ 佇列中的訂單相關事件的消費者。它負責監聽佇列內的消息，並根據不同的事件類型，執行對應的處理邏輯。

3️⃣ 事件處理流程

消費者會透過 consume_messages() 方法來處理不同的訂單事件：

  • 訂單創建事件（Order Created）
    • 消費者會從 ORDER_CREATED_QUEUE 接收消息，並執行：
      • 發送訂單確認郵件
  • 支付處理事件（Payment Processed）
    • 消費者會從 PAYMENT_PROCESSED_QUEUE 接收消息，並執行：
      • 驗證支付狀態
  • 訂單發貨事件（Order Shipped）
    • 消費者會從 ORDER_SHIPPED_QUEUE 接收消息，並執行：
      • 處理物流狀態更新
