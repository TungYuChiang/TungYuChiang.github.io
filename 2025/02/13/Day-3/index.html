Model vs Schema

Model（SQLAlchemy ORM 模型） → 負責資料庫

👉 Model 主要是跟資料庫打交道，對應到資料庫表（table）
tips : 可以理解成，Table 在fastAPI以一個class的方式存在

Schema（Pydantic 模型） → 負責 API 數據驗證

👉 Schema 主要用來驗證 API 的輸入/輸出數據，讓 FastAPI 確保請求數據正確！

流程

1. 使用 Schema（Pydantic）驗證 API 請求

  • 使用 Schema（Pydantic）驗證 API 請求，並呼叫對應的service
  • ProductCreate：確保 API 的請求數據正確。
  • response_model=ProductSchema：確保回應時符合 ProductSchema，可以轉成 JSON。

2. 使用 Service（SQLAlchemy ORM）存入資料庫

  • ProductCreate 是 API 傳來的數據
  • 轉換成 Product ORM 物件後，存入 PostgreSQL
  • 回傳 Product ORM 物件（但 orm_mode = True 讓它可以轉 JSON）

FastAPI IoC & Dependency Injection

1️⃣ 什麼是 IoC（控制反轉）？

IoC（Inversion of Control） 是一種設計模式，它的核心概念是：
✅ 程式的控制權從手動管理，交給框架來處理。
✅ 降低模組之間的耦合度，提高靈活性。
✅ 由框架負責創建 & 傳遞依賴物件，而不是手動初始化。
在傳統寫法中，我們會這樣手動管理資料庫 Session：

這種做法雖然直觀，但：

  • 需要手動管理 db 連線
  • 每個函數都要寫 db = SessionLocal()<
  • 不能自動回收 db

2️⃣ FastAPI 的 Depends(get_db) 如何應用 IoC？

FastAPI 的 Depends(get_db) 是一種 依賴注入（Dependency Injection, DI），它允許：
✅ FastAPI 自動管理 db，不需要手動初始化
✅ 確保 db 連線在請求結束後自動關閉
✅ 提升程式的可讀性與可維護性

🔹 get_db() 提供 Session

這裡的 yield db 讓 FastAPI 管理 db 的生命週期，這就是 IoC（控制反轉） 的概念。

🔹 Depends(get_db) 自動注入 db

這邊的 Depends(get_db)：

  1. FastAPI 會自動呼叫 get_db()
  2. yield db 會把 Session 傳入 db 參數
  3. 請求結束後，FastAPI 自動關閉 db

這樣開發者不需要手動管理 db。
